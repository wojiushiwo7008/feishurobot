二、目标与约束
目标：租户邮件投诉“水管漏水”后，系统能自动完成“识别→派单→预约→维修→回访→结案→沉淀数据”，尽量减少人工介入，同时保证合规、及时、可追溯。
关键约束：
紧急程度判断（漏水可能是紧急维修，影响财产/安全）
SLA（例如：紧急 15 分钟内响应；非紧急 4 小时内响应）
权限与合规（隐私、房东授权、费用审批、沟通留痕）
异常兜底（无法解析邮件、租户不配合、供应商爽约等）
1) 触发与收信入口
渠道：租赁管理专用邮箱
收信机制：
用邮件收发服务的 Webhook/IMAP Polling 把新邮件推到系统（建议 webhook + 重试队列）
每封邮件生成一个 InboundEmail 事件，写入事件表/消息队列（Kafka/SQS/RabbitMQ 均可）
基础反垃圾/安全：
SPF/DKIM/DMARC 验证结果记录
附件安全扫描（图片/视频/文档），可疑附件隔离
2) 邮件解析与“工单”创建（核心自动化第一步）
2.1 解析邮件内容
从邮件中提取：
发件人邮箱、姓名（如果签名里有）
物业地址/房号（常见：租户会写“我住在XX”）
问题描述（漏水位置：厨房/浴室/主水管；是否有停水；是否影响用电）
时间信息（什么时候开始漏、是否持续）
附件（照片/视频）
2.2 租户与物业匹配
用发件人邮箱/电话去租赁系统（Property / Tenancy / Lease）里匹配：
tenant_id、property_id、lease_id
若匹配失败：进入“人工补全”队列，并自动回复租户补充信息（见兜底）
2.3 意图识别 + 分类（AI 参与点）
用 LLM/分类模型做：
意图：维修请求（Maintenance）
类别：Plumbing / Leak
紧急等级：Emergency / Urgent / Normal
结构化字段（location、severity、access_preference、preferred_time）
并输出一个“可解释”的结果（比如：判定紧急的依据：“持续漏水 + 已渗到地板 + 担心电器”）。
2.4 创建工单（Ticket）
在租赁管理系统创建：
ticket_id
roperty_id、tenant_id
category=Plumbing, issue=Leak
priority=Emergency（示例）
SLA_deadline（按优先级计算）
状态：NEW -> TRIAGED
同时把原邮件内容、附件链接、模型输出、每一步动作写入 审计日志。
3) 自动回复租户：确认已受理 + 收集关键信息
T+1 分钟内自动回信（这是体验与合规的关键）：
邮件内容包含：
已受理：工单号、预计响应时间（基于 SLA）
安全提示（非常专业）：
关闭总阀/局部阀门（如果可行）
漏水接近电器/插座请先断电
如发生严重渗漏可先联系紧急电话（提供号码）
需要补充的信息（如果缺失）：
漏水位置、是否持续、是否影响邻居/楼下
是否允许维修人员进入（是否有宠物、是否需要提前短信）
可预约时间段（比如 9-12 / 12-3 / 3-6）
如果信息齐全，邮件就不问太多，直接进入派单流程；信息不全才追问，减少往返。
4) 规则引擎 + SLA 路由（决定“怎么处理”）
这里你可以强调：AI 决策 + 规则兜底。
4.1 紧急维修规则（Plumbing Leak 常见）
若满足任一条件 → Emergency：
持续漏水且无法止住
影响供电/有触电风险
渗漏到邻居/公共区域
可能造成财产重大损失
Emergency 流程：
立即派发给“紧急合作水管工”池
同时短信/邮件通知租赁经理（可选：电话机器人）
若超时未接单 → 自动升级到第二供应商/值班经理
4.2 非紧急流程
可在工作日时间段预约
需要房东报价审批的，先走审批（见第 6 节）
5) 供应商派单与预约（自动化第二核心）
5.1 供应商选择
从 Vendor DB 选择：
地理位置/覆盖范围
可用时间（Calendar）
价格等级/合同条款
历史评分（准时率、一次修复率、投诉率）
是否具备紧急出勤能力
可以说一句很加分的话：
“我会做一个 Vendor Ranking：综合距离、响应时间预测、历史KPI、费用上限，输出 Top3 并自动触达。”
5.2 自动触达供应商（邮件为主）
给供应商发“派单邮件”包含：
工单号、地址（注意隐私：仅必要字段）
问题描述 + 照片链接
紧急等级 + SLA
期望到场时间窗口
费用规则（如：紧急 call-out fee 上限）
确认方式：让供应商 “回复邮件 / 点击链接” 进行接单与选时间
5.3 邮件回执处理
系统监听供应商回复：
若回复“接受 + 时间” → 更新 ticket 状态 SCHEDULED，同步给租户确认
若拒绝/无响应超过阈值 → 自动发给下一个供应商并升级告警
6) 费用与审批
漏水有时涉及较高费用或需要房东授权。你可以设计：
规则：
<= $X 可自动批准（依据管理协议/紧急维修条款）
> $X 需房东审批
审批触达：给房东发邮件（或 portal 通知），包含：
问题摘要（AI 生成短摘要）
报价/预估费用
风险提示（不修的后果）
1-click 批准/拒绝链接
超时策略：
紧急维修：若合同允许“紧急先修后报”，则先行动并事后通知；否则升级人工电话跟进。
7) 现场执行与状态更新
维修到场后，供应商可以通过邮件回传：
已到场 / 已完成 / 需要追加零件 / 追加报价
系统解析后更新状态：
IN_PROGRESS -> WAITING_APPROVAL -> COMPLETED
并同步通知租户：
“预计完成时间”
“是否需要二次上门”
“完成后请确认是否解决”
8)结案、回访与知识沉淀
8.1 租户确认 & 自动回访
维修完成后自动发邮件：
请租户确认是否修复
评分（1-5）+ 可选文字反馈
若 48 小时未确认：自动提醒；仍未确认可按规则自动结案但标记“未确认结案”
8.2 工单关闭与归档
关闭条件：
供应商完成报告 + 租户确认（或超时规则）
费用单据入账（invoice）关联到 property/owner statement
8.3 数据沉淀
漏水原因标签（老化/堵塞/人为）
修复方式与成本
供应商 KPI（响应时长、一次修复率）
常见问题知识库更新（用于未来自动回复更准）
9)异常与兜底
无法匹配租户/地址：自动回信要求补充地址 + 电话；同时进入人工队列
模型不确定（低置信度）：走规则引擎或人工审核
供应商超时不响应：自动 escalations（Top3轮询 + 值班经理告警）
租户不允许入内/失联：记录“Access blocked”，暂停 SLA 并通知租赁经理
重复邮件/同一问题多次投诉：基于 thread_id / similarity 去重合并工单
高风险内容（例如“天花板塌了/煤气味”）：直接触发紧急电话提示 + 人工介入
10) 总结：邮件入口：Webhook/IMAP → Message Queue
核心服务：Parser + LLM Triage Agent + Rules Engine + Ticket Service
集成：Property/Lease DB、Vendor DB、Calendar、SMS/Email、审批系统、账单系统
可观测性：全链路 trace + 审计日志 + SLA dashboard
安全合规：PII 最小化、权限分级、附件扫描、可回放的操作记录

三、merge 是通过生成新的 merge commit 保留分叉历史，不会改写历史；
rebase 是把当前分支的提交重放到目标分支之后，生成新的提交，历史更线性但会改写 hash。
我在个人 feature 分支开发阶段使用 rebase 同步主分支，在公共分支合并时使用 merge，避免重写历史影响团队

四、当我接到完全陌生的技术需求时，我不会直接写代码，我会先明确业务目标和成功标准，然后抽象出系统的输入输出，建立最小认知模型。接着快速扫描文档和现有代码，找到核心控制点，做一个最小可运行验证来降低不确定性。最后再拆解任务和评估风险。我的原则是先减少不确定性，而不是追求一次性设计完美方案。
